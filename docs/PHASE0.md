# Phase 0: what is built, and what comes next

This scaffold gives the build agent running code to extend rather than a blank repo. Phases 0 through 3 of the design doc are built, plus the leech section from Phase 5: a working register, lessons, and reviews loop with the on-screen Cyrillic keyboard, automatic level progression, a progress dashboard, and a dedicated leech section with no-stakes practice, all on a fully tested SRS engine. The main remaining work is content at scale and offline/sync.

## Built and verified

- **Stable item identity and content importer** (`backend/app/content/`): items now carry a unique `external_id`; all content loads through `upsert_items`, which upserts on that key so refreshing or extending the deck never reassigns row ids or orphans user progress. Migration `3284f0181e9e` adds it safely on both empty and populated databases (nullable, backfill, then NOT NULL via batch mode). The seed was refactored to use the importer. 5 importer tests, 174 backend tests total. Repo now ships `CLAUDE.md` (contributor and Claude Code guide) and `docs/PRODUCTION_READINESS.md` (the sequenced build guide with per-task acceptance criteria) for porting to GitHub.

- **SRS engine** (`backend/app/srs/engine.py`): stage ladder, standard and early-level-accelerated intervals, the WaniKani decrement formula, scheduling, level-gating thresholds, and leech scoring. 82 passing tests in `backend/tests/test_srs_engine.py`. This is pure and dependency-free on purpose.
- **Auth** (`backend/app/routers/auth.py`): register, login, current-user, with Argon2 hashing and JWT. Smoke-tested end to end.
- **Schema** (`backend/app/models.py`): all tables from design doc section 17 (users, items, example_sentences, mnemonics, user_item_state, review_events, lesson_events, push_subscriptions, audio_assets).
- **App wiring** (`backend/app/main.py`): CORS for the Vite dev server, health check, dev table creation, Alembic configured for real migrations.
- **Lessons and reviews loop** (`backend/app/services/learning.py`, routers `lessons.py` and `reviews.py`): the engine is now wired to live endpoints. `GET/POST /lessons`, `GET/POST /reviews` with WaniKani-style pass tracking (an item advances only when every required question type has a correct answer; any miss in the pass still lowers it), idempotent submits, and a dev seed of real Russian words. 22 tests covering the flow and grading.
- **Grading** (`backend/app/grading.py`): server-side, stress-insensitive Russian matching, ё/е equivalence, English accept-lists with article stripping, typo tolerance, and the correct / near-miss / incorrect outcomes.
- **Level progression and dashboard** (`backend/app/services/dashboard.py`, level-up logic in `learning.py`): the next level unlocks automatically when the current one crosses its Guru threshold, and `GET /dashboard` returns level progress, SRS-stage counts, due and next-24h review counts, streak, and accuracy. The home screen renders all of it, and review feedback shows a level-up banner.
- **Leech section** (`backend/app/services/study.py`, `frontend/src/components/LeechesPage.tsx`): `GET /leeches` with stats, a dedicated leech page, and `POST /leeches/study` for a leech-only session. Practice runs through `POST /practice`, which grades with no SRS side effects, so cramming a leech does not advance it. Also includes `GET /extra-study` (recent mistakes, recently learned, by level) and `PUT /items/{id}/mnemonic` for per-word mnemonics editable from the leech page. 117 backend tests total.
- **On-screen Cyrillic keyboard** (`frontend/src/components/CyrillicKeyboard.tsx`): JCUKEN default with a phonetic toggle and long-press letter hints. Typechecks and builds.
- **Frontend app** (`frontend/src/`): a full Phase 2 UI in Vite + React + TS + Tailwind. Auth screen, a home dashboard with live lesson and review counts, a lesson walkthrough, and a review session that drives meaning and production reviews (production via the Cyrillic keyboard), with grading feedback, the stressed form shown on every answer, near-miss retry with override, and client-side re-quiz of misses. `npm run build` passes.
- **Production hardening (P0 code batch complete, per `PATH-TO-PAID.md`)**:
  - Auth: refresh tokens with rotation and revocation, logout-all, email verification, password reset, per-IP rate limiting; config fails fast on a default JWT secret in production.
  - Billing and entitlements: a `Subscription` model, a free-levels paywall enforced server-side in lessons and level-up (free users are walled at `free_level_limit`, reviews stay ungated), and a Stripe-ready scaffold (`POST /billing/checkout`, `/portal`, `/webhook`, `GET /billing/status`). The webhook event handler that syncs entitlement is tested with synthetic events; live Stripe needs only keys and price IDs.
  - Performance: `get_reviews` batch-loads items and events, the dashboard uses SQL aggregates, and the lesson cap is a SQL count.
  - Migrations: an initial Alembic migration (`app/migrations/versions/`) creates the full schema and applies cleanly; production no longer auto-creates tables (`create_all` is dev-only).
  - 134 backend tests. Remaining P0 work needs your accounts or assets: live Stripe keys, a real email provider, a Redis-backed limiter, deployment and CI, the content pipeline, and the legal/licensing review.
- **Sync, vacation, and settings** (`backend/app/services/account.py`, sync in `learning.py`): `POST /reviews/sync` replays a batch of offline review answers idempotently (re-syncing is safe), `POST /settings/vacation` freezes and unfreezes the schedule (shifting `available_at` forward on return so nothing is overdue), and `GET`/`PATCH /settings` manage the daily lesson cap, autoplay, and keyboard layout. 138 backend tests.
- **End-of-session summary**: reviews and lessons now end on a recap screen (shared `SessionSummary` component) instead of a one-line message. Reviews show items reviewed, first-attempt accuracy, and answer count, with celebratory highlights for any level-up, items that reached Guru, and items burned, plus the offline-pending note when relevant. Lessons show words learned and quiz accuracy. Outcomes are accumulated during the session from the existing review/grade responses (no backend change) and read at the end. Frontend typechecks and builds.
- **Burned-item resurrection and a stats page**: burned (retired) items can now be resurrected back into the queue at Apprentice 1, due immediately, completing the SRS lifecycle (`GET /burned`, `POST /items/{id}/resurrect`, 5 tests; a Burned-items page plus a Resurrect button in item detail). A progress stats page (`GET /stats`, 3 tests) surfaces totals (reviews, accuracy, current and longest streak, words started and burned), a 30-day reviews-per-day chart with the correct portion highlighted, and the SRS distribution, all derived from the review-event log. Frontend typechecks and builds.
- **User synonyms**: learners can add their own accepted meanings for a word (e.g. accept "town" for "city"), so a correct-but-unlisted answer stops being marked wrong, the most common review frustration. Synonyms are per-user-per-item (`user_synonyms` table, third Alembic migration), merged into the meaning accept-list during server-side review grading, deduped case-insensitively and capped per item. Managed from the item detail view (add/remove chips), and a one-tap "accept this next time" affordance appears in the review feedback when a meaning answer is marked wrong. Backend 7 tests; frontend typechecks and builds.
- **Testing and performance hardening**: the frontend now has a Vitest suite (jsdom + fake-indexeddb), 26 tests covering the grading port (pinned against the server rules), the IndexedDB offline queue, and the sync-drain logic, so the critical client logic is no longer only typecheck-verified. On the backend, the duplicated UTC/aware time helpers were consolidated into `app/timeutil.py`, and composite indexes were added on the hot query paths (`user_item_state(user_id, available_at)`, `review_events(user_id, answered_at)` and `(user_id, item_id)`, `lesson_events(user_id, learned_at)`) with a second Alembic migration that applies cleanly on top of the initial schema. A stray build-artifact emission (`vite.config.js`/`.d.ts` leaking into the source tree on every build) was fixed by redirecting the composite project's emit. 161 backend tests, 26 frontend tests.
- **Frontend, expanded UI** (`frontend/src/`): on top of the core loop, the app now has a token-refresh auth layer (access plus refresh in storage, a single-flight auto-refresh on 401 with retry, logout that revokes the refresh token, and a forgot/reset-password flow), an **item browser** (search, level and part-of-speech filters, paginated list with SRS-status badges, and a detail view with examples, mnemonic, and progress), a **settings** page (daily lesson cap, autoplay, keyboard layout) with the **vacation toggle**, and an **upgrade/billing** screen (plan choices, Stripe checkout and portal redirects, premium status). All typecheck and `npm run build` passes; not runtime-tested here (no browser).
- **End-of-lesson quiz (client-side)**: faithful to WaniKani, the lesson quiz runs entirely in the browser so there are no per-answer network round-trips while drilling. Grading rules are ported to TypeScript (`frontend/src/lib/grading.ts`) and verified byte-identical to the server across stress-mark stripping, ё/е equivalence, typo-tolerance boundaries, and article handling. The learner browses the new words, drills meaning and production with retries on misses, then the passed batch is committed in a single call. The commit endpoint (`POST /lessons/complete`) still enforces the daily cap and the paywall server-side so the gate cannot be bypassed, returning started / over_cap / skipped per item.
- **Item browser** (`GET /items`, `GET /items/{id}`): browse and search the vocabulary with level and part-of-speech filters and pagination, each row annotated with the user's SRS status (locked / available / band) and entitlement. Detail returns forms, example sentences, the user's mnemonic, and full per-item state. Reads existing tables only, so the migration is unchanged. 13 tests.
- **Data spike** (`pipeline/spike_data_check.py`): validates the content foundation before the deck is built.

## Stubbed or not yet built (by phase, per the design doc)

- **Phase 1, content pipeline:** only the spike exists. Build the full pipeline (frequency join, stress resolution, audio fetch, TTS generation, level bucketing, seed emit + validation) and a seed step that loads it into Postgres. This is the critical path; run the spike first.
- **Phase 2, lessons and reviews: DONE.** Backend loop, grading, the review UI, and the Cyrillic keyboard are built and tested. One refinement remains for later: a grade-then-commit split so override can also waive a hard-wrong answer, not just a near-miss.
- **Phase 3, gating and dashboard: DONE.** Levels now advance automatically when the current one clears its Guru threshold, and the dashboard (endpoint plus the home-screen UI) shows level progress, SRS-stage counts, forecast, streak, and accuracy.
- **Phase 4, offline and sync: now built (frontend not runtime-tested).** The backend replay endpoint (`POST /reviews/sync`) was already tested. The frontend now has a hand-rolled service worker that caches the app shell (offline reloads), an IndexedDB queue (`lib/offlineQueue.ts`) that captures review answers when a submit fails or the device is offline, and a sync manager (`lib/sync.ts`) that drains the queue into `/reviews/sync` on reconnect and at startup. Reviews never ship the answer, so offline answers are queued and graded server-side on sync rather than locally (preserving review integrity); the UI shows a "saved offline, will sync" state. Web push is wired end to end on the client (`lib/push.ts`, an Enable-reminders button in Settings) with a backend store for subscriptions (`POST /push/subscribe`, `DELETE /push/subscribe`, 4 tests) and a push handler in the service worker; actual delivery (a scheduler plus VAPID keys) is server infrastructure and intentionally left as a stub.
- **Phase 5, leeches, extra study, vacation: frontend now complete.** The settings page, vacation toggle, and an extra-study launcher (recent mistakes, recently learned, or by level, all no-stakes practice) are wired in. Nothing material remains on the frontend for this phase.
- **Phase 6, self-add and alphabet primer:** lookup auto-fill, on-demand TTS endpoint, level 0.
- **Phase 7, polish and full seed.**

## Conventions for the build agent

- Keep the SRS engine pure. Wrap it in services that read and write `user_item_state` and append to `review_events`.
- Treat `review_events` as the append-only source of truth; `user_item_state` is a derived projection.
- All new state is keyed by `user_id` (multi-user schema, single user for now).
- No em dashes in user-facing copy (author preference).
- Python for data work; tests alongside the correctness-critical logic.
